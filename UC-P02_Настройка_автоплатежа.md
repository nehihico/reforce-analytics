# UC-P02: Настроить автоплатёж по договору

**Версия:** 1.0 | **Дата:** 2026-06-01 | **Статус:** Черновик  
**Итерация:** 1 (текущая) | **Применимо:** все типы клиентов

| | |
|---|---|
| **Краткое описание** | Система автоматически списывает абонентскую плату (и накопленный долг при наличии) в дату начисления по договору, используя сохранённый платёжный метод клиента |
| **Триггер** | Наступила дата начисления абонентской платы по договору, для которого активен автоплатёж |

| **Предусловия** | |
|---|---|
| | • Автоплатёж активирован по договору (см. UC-P01) |
| | • `payment_method_id` сохранён в системе |
| | • Договор активен, начислена абонентская плата |
| | • ЮКасса доступна |

| **Участники** | |
|---|---|
| | • Система ЛК (cron-задача) |
| | • ЮКасса |
| | • Клиент (получает уведомления) |

---

## Основной поток (успешное списание)

| Шаг | Действие | Ответ системы | Макеты |
|-----|----------|--------------|--------|
| 1 | За 3 дня до даты списания cron-задача формирует уведомление | Система отправляет пуш: «Через 3 дня спишем {сумма} ₽ по договору №{N} (СБП / карта ••••1234)» | Прил. В.7 |
| 2 | В день списания cron-задача инициирует рекуррентный платёж | Система создаёт платёж в ЮКассе: `payment_method_id: {сохранённый ID}`, `amount`: абонентская плата [+ долг, если есть], `capture: true` | |
| 3 | ЮКасса проводит списание | Вебхук `payment.succeeded` → бэкенд ЛК | |
| 4 | Бэкенд обрабатывает вебхук | Система обновляет баланс договора. Отправляет пуш: «Списано {сумма} ₽. Автоплатёж по договору №{N}. Баланс: {баланс} ₽» | |

---

## Альтернативный поток A: долг на момент списания

| Шаг | Условие | Действие системы | Ответ |
|-----|---------|-----------------|-------|
| 1а | На момент даты начисления у клиента есть накопленный долг по этому договору | Cron за 3 дня отправляет расширенный пуш | «Через 3 дня спишем {абон. плата + долг} ₽ по договору №{N}. Включает задолженность {долг} ₽» |
| 2а | В день списания | Система создаёт платёж на сумму абон. плата + долг | Дальше — основной поток шаги 3–4 |

---

## Альтернативный поток B: фоллбэк СБП → карта

| Шаг | Условие | Действие системы | Ответ |
|-----|---------|-----------------|-------|
| 2б | Списание через СБП завершилось неудачей | Система немедленно инициирует повторную попытку списания через сохранённый токен карты | |
| 3б | Списание по карте прошло успешно | Вебхук `payment.succeeded` → бэкенд | Пуш: «Списано {сумма} ₽ с карты •••• 1234 (списание через СБП не прошло) по договору №{N}» |
| 3в | Списание по карте также неудачно | — | Переход к потоку C (повторные попытки) |

> Открытый вопрос: фоллбэк происходит в ту же попытку (мгновенно) или через 3 дня как следующая попытка? Требует уточнения у заказчика.

---

## Альтернативный поток C: все попытки неудачны

| Шаг | Условие | Действие системы | Ответ |
|-----|---------|-----------------|-------|
| — | 1-я попытка не прошла | Система ждёт 3 дня, отправляет пуш: «Не удалось списать {сумма} ₽. Оплатите вручную или обновите способ оплаты» | |
| — | 2-я попытка (день +3) не прошла | Повторная попытка. Пуш аналогичный | |
| — | 3-я попытка (день +6) не прошла | Автоплатёж приостанавливается. Пуш: «Автоплатёж по договору №{N} приостановлен: 3 попытки списания не прошли. Оплатите вручную» | |

---

## Правила формирования суммы списания

| Ситуация | Сумма списания |
|---|---|
| Нет долга | Абонентская плата за 1 месяц |
| Есть долг | Абонентская плата + полный долг |
| Долг превышает абон. плату | Пуш за 3 дня содержит развёрнутое объяснение суммы |

---

| **Постусловия** | |
|---|---|
| Успешный сценарий | Баланс договора обновлён. Клиент получил пуш-подтверждение |
| Неуспешный сценарий (3 попытки) | Автоплатёж приостановлен. Долг остаётся. Клиент уведомлён |

| **Нефункциональные требования** | |
|---|---|
| Идемпотентность | Каждый запрос создания платежа содержит уникальный `Idempotence-Key`, защита от двойного списания |
| Обновление баланса | При получении `payment.succeeded` баланс обновляется немедленно, не ожидая синхронизации с 1С |
| Логирование | Все операции логируются для разбора спорных ситуаций |

| **Связанные UC** | UC-P01 (Привязка метода), UC-P03 (Управление методом), UC-P05 (Логика после сбоя) |

## Критерии приёмки (Acceptance Criteria)

### AC-P02-01: Пуш-уведомление за 3 дня до списания

**Given** автоплатёж активен по договору, дата списания через 3 дня  
**When** cron-задача выполняется  
**Then**
- клиент получает пуш: «Через 3 дня спишем {сумма} ₽ по договору №{N}»
- если есть накопленный долг — сумма в пуше включает долг, текст поясняет: «Включает задолженность {долг} ₽»
- пуш отправляется в интервале 08:00–21:00 по местному времени клиента

---

### AC-P02-02: Успешное автосписание

**Given** наступила дата списания, автоплатёж активен  
**When** ЮКасса проводит рекуррентный платёж  
**Then**
- баланс договора обновляется немедленно после получения вебхука `payment.succeeded` (не ожидая синхронизации с 1С)
- клиент получает пуш: «Списано {сумма} ₽. Автоплатёж по договору №{N}. Баланс: {баланс} ₽»
- в истории операций договора появляется запись о списании

---

### AC-P02-03: Списание при наличии долга

**Given** на момент даты начисления у клиента есть долг по договору  
**When** cron инициирует рекуррентный платёж  
**Then**
- сумма платежа = абонентская плата + накопленный долг
- пуш за 3 дня содержит расшифровку суммы
- после успешного списания баланс договора становится 0 (долг + текущий месяц погашены)

---

### AC-P02-04: Двойное списание исключено

**Given** cron-задача сработала дважды (технический сбой)  
**When** оба запроса уходят в ЮКассу с одинаковым `Idempotence-Key`  
**Then**
- ЮКасса проводит платёж только один раз
- на стороне ЛК фиксируется только одна транзакция

---

### AC-P02-05: Пуш не отправляется вне допустимого окна

**Given** расчётное время отправки пуша — 22:30 по местному времени клиента  
**When** cron формирует очередь уведомлений  
**Then**
- пуш откладывается до 08:00 следующего дня
- списание происходит в штатное время независимо от времени пуша
