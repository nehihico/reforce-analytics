# UC-P05: Обработать сбой автоплатежа (недостаток средств и другие ошибки)

**Версия:** 1.0 | **Дата:** 2026-06-01 | **Статус:** Черновик  
**Итерация:** 1 (текущая) | **Применимо:** все типы клиентов

| | |
|---|---|
| **Краткое описание** | Система обрабатывает ситуацию, когда автоматическое списание не прошло (недостаток средств, блокировка карты, ошибка банка, технический сбой). Система предпринимает повторные попытки, при исчерпании — приостанавливает автоплатёж и уведомляет клиента |
| **Триггер** | ЮКасса вернула статус `payment.canceled` или `payment.failed` на запрос рекуррентного платежа |

| **Предусловия** | |
|---|---|
| | • Автоплатёж активирован по договору |
| | • Первая попытка автосписания завершилась неудачей |
| | • `payment_method_id` (СБП или карта) сохранён |

| **Участники** | |
|---|---|
| | • Система ЛК (cron-задача) |
| | • ЮКасса |
| | • Клиент (получает уведомления) |

---

## Классификация причин сбоя

| Код ошибки ЮКассы | Причина | Стратегия |
|---|---|---|
| `insufficient_funds` | Недостаток средств | Повторить через 3 дня |
| `card_expired` | Карта истекла | Немедленно уведомить, предложить сменить метод |
| `do_not_honor` | Банк отклонил | Повторить через 3 дня |
| `invalid_card` | Карта заблокирована | Немедленно уведомить, предложить сменить метод |
| `timeout` / технический сбой | Сбой на стороне ЮКассы | Повторить через 1 час (технический retry), не считать как «неудачную попытку» клиента |

> Требует уточнения с бэкендом: разграничение между техническим retry (без уведомления клиента) и «официальной» повторной попыткой (с уведомлением) нужно согласовать с командой разработки.

---

## Основной поток — два метода активны (СБП + карта, фоллбэк)

> Решение: фоллбэк происходит мгновенно (в ту же попытку). Зависит от поддержки ЮКассой — уточнить с бэкендом на этапе реализации.

| Шаг | Действие | Ответ системы | Макеты |
|-----|----------|--------------|--------|
| 1 | ЮКасса возвращает неудачу по СБП-методу | Система немедленно инициирует попытку через `payment_method_id` карты (без ожидания 3 дней) | |
| 2а | Списание по карте прошло | Вебхук `payment.succeeded` → бэкенд. Баланс обновлён | |
| 3 | Клиент получает пуш | «Списано {сумма} ₽ с карты •••• 1234 (списание через СБП не прошло) по договору №{N}» | Прил. В.7 |

---

## Основной поток — один метод, повторные попытки

| Шаг | Действие | Ответ системы | День |
|-----|----------|--------------|------|
| 1 | 1-я попытка. Неудача | Пуш: «Не удалось списать {сумма} ₽ по договору №{N}. Оплатите вручную или обновите способ оплаты →» | День 0 |
| 2 | Клиент не оплатил вручную. Система ждёт 3 дня | — | День 0–3 |
| 3 | 2-я попытка. Система инициирует повторное списание | — | День 3 |
| 4а | 2-я попытка успешна | Пуш: «Списано {сумма} ₽. Автоплатёж по договору №{N}» | День 3 |
| 4б | 2-я попытка неудачна | Пуш: «Не удалось списать {сумма} ₽. Осталась 1 попытка» | День 3 |
| 5 | 3-я (последняя) попытка через 3 дня | — | День 6 |
| 6а | 3-я попытка успешна | Пуш об успехе | День 6 |
| 6б | 3-я попытка неудачна | Автоплатёж приостанавливается. Пуш: «Автоплатёж по договору №{N} приостановлен: 3 попытки не прошли. Оплатите вручную или смените способ оплаты →» | День 6 |

---

## Поведение клиента во время периода повторных попыток

| Действие клиента | Реакция системы |
|---|---|
| Оплатил вручную | Долг погашен. Следующий автосписание — в штатном режиме. Счётчик попыток сбрасывается |
| Сменил платёжный метод (UC-P03) | Система немедленно инициирует новую попытку с новым методом. Счётчик сбрасывается |
| Пополнил счёт (если причина — недостаток средств по СБП) | При следующей плановой попытке списание пройдёт автоматически |
| Ничего не делает | Система исчерпывает попытки → автоплатёж приостанавливается |

---

## Состояние после приостановки автоплатежа

| Параметр | Значение |
|---|---|
| Статус автоплатежа | `suspended` (приостановлен, не `disabled`) |
| `payment_method_id` | Сохранён |
| Повторная активация | Клиент может активировать автоплатёж вручную (UC-P01), при этом должен ли он заново привязывать метод — см. открытый вопрос |
| Пуш-напоминания о долге | Продолжают отправляться по таблице эскалации (FR-4.1.1) |
| Баннер долга | Отображается на главном экране |

## Поток: успешный ручной платёж во время сбоя (контекстное предложение)

| Шаг | Действие | Ответ системы |
|-----|----------|--------------|
| 1 | Клиент оплатил вручную, получил экран «Спасибо! Платёж принят» | Система отображает блок: «Подключите автоплатёж, чтобы не забывать» (кнопки «Подключить» / «Нет, спасибо») |

---

| **Постусловия** | |
|---|---|
| Успешный сценарий (фоллбэк или повторная попытка) | Долг погашен. Баланс обновлён. Клиент уведомлён. Автоплатёж остаётся активным |
| Неуспешный сценарий (3 попытки) | Автоплатёж приостановлен. Долг остаётся. Клиент уведомлён. Пуши о долге продолжаются |

| **Нефункциональные требования** | |
|---|---|
| Идемпотентность | Повторные попытки используют уникальный `Idempotence-Key`, исключающий двойное списание |
| Логирование | Все попытки и их результаты логируются (для разбора споров) |
| Обновление баланса | При успехе любой попытки баланс обновляется немедленно (по вебхуку), без ожидания синхронизации с 1С |

| **Связанные UC** | UC-P01 (Привязка метода), UC-P02 (Автоплатёж), UC-P03 (Управление методом), UC-P04 (Отмена) |
|---|---|
| **Связанные требования** | FR-4.2.3, FR-5.1, NFR-6 |
| **Решения** | Фоллбэк СБП→карта — мгновенно (уточнить поддержку в ЮКассе с бэкендом) |
| **Открытые вопросы** | Технический retry (timeout ЮКассы): считается официальной попыткой? После приостановки: нужна новая привязка или переиспользуем токен? |

## Критерии приёмки (Acceptance Criteria)

### AC-P05-01: Мгновенный фоллбэк СБП → карта

**Given** у клиента активны два метода: СБП и карта  
**When** списание через СБП отклонено ЮКассой  
**Then**
- система немедленно инициирует повторную попытку через токен карты (без ожидания 3 дней)
- при успехе клиент получает пуш: «Списано {сумма} ₽ с карты •••• 1234 (списание через СБП не прошло) по договору №{N}»
- счётчик неудачных попыток не увеличивается (фоллбэк не считается отдельной попыткой)

---

### AC-P05-02: Уведомление после первой неудачной попытки

**Given** единственный платёжный метод, списание не прошло  
**When** ЮКасса возвращает ошибку  
**Then**
- клиент получает пуш: «Не удалось списать {сумма} ₽ по договору №{N}. Оплатите вручную или обновите способ оплаты»
- пуш содержит deep link на экран оплаты договора
- счётчик попыток = 1, следующая попытка через 3 дня

---

### AC-P05-03: Повторная попытка через 3 дня

**Given** первая попытка не прошла, клиент не оплатил вручную  
**When** прошло 3 дня  
**Then**
- система автоматически инициирует вторую попытку списания
- при успехе — пуш об успешном списании, счётчик сбрасывается
- при неудаче — пуш: «Не удалось списать {сумма} ₽. Осталась 1 попытка»

---

### AC-P05-04: Приостановка после трёх неудач

**Given** три попытки подряд завершились неудачей  
**When** третья попытка отклонена  
**Then**
- автоплатёж переходит в статус `suspended`
- клиент получает пуш: «Автоплатёж по договору №{N} приостановлен: 3 попытки не прошли. Оплатите вручную»
- пуш-напоминания о долге по таблице эскалации продолжают приходить
- баннер долга на главном экране сохраняется

---

### AC-P05-05: Ручная оплата сбрасывает счётчик попыток

**Given** первая или вторая попытка не прошла, клиент оплатил вручную  
**When** вебхук `payment.succeeded` получен бэкендом  
**Then**
- счётчик неудачных попыток сбрасывается до 0
- следующий автоплатёж выполняется в штатном режиме в дату следующего начисления
- после успешной ручной оплаты показывается предложение подключить/оставить автоплатёж

---

### AC-P05-06: Смена метода сбрасывает счётчик и инициирует новую попытку

**Given** клиент сменил платёжный метод во время периода повторных попыток  
**When** новый `payment_method_id` сохранён  
**Then**
- система немедленно инициирует попытку списания с новым методом
- счётчик попыток сбрасывается
- при успехе — штатный пуш об успешном списании

---

### AC-P05-07: Двойное списание исключено при retry

**Given** система инициирует повторную попытку  
**When** первый запрос завис и система отправила второй с тем же `Idempotence-Key`  
**Then**
- ЮКасса обрабатывает платёж только один раз
- на стороне ЛК фиксируется одна транзакция, баланс обновляется корректно
